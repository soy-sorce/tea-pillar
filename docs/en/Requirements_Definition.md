## 1. Project Overview

**Product name:** nekkoflix  
**Concept name:** Pets in the Loop  
**Form:** Web application (hackathon MVP)

### One-line summary

A Web app that uses multimodal AI to analyze a cat's state, including meows, facial expressions, and pose, then generates and delivers a video optimized for that cat in real time. The cat's reaction data is fed back into the model, forming a loop in which "cats help train AI."

### Mission

> "Humans and cats are teaching AI together."  
> Before Physical AI becomes commonplace, we provide a Web-based preview of how cats might first encounter AI.

### Story

We began by thinking only about human-centered AI products, then realized we had completely forgotten animals. For humans, AI is already becoming familiar. For animals, however, AI may still be entirely new. Our response to the hackathon theme is that AI is still "Brand New" for animals, and this product creates the moment when a cat first meets AI.

## 2. Background and Objectives

### Background

- Physical AI and robotics are expected to become common in households within 10 to 20 years.
- Current AI development is dominated by "Human in the Loop," with almost no consideration of the animal perspective.
- Our interpretation of the hackathon theme is that AI is still a "Brand New" existence for animals.

### Objectives

1. Demonstrate the new concept of cat-specific AI entertainment as a prototype.
2. Convey the concept intuitively to hackathon judges through an experience where they "become the cat."
3. Accumulate cat reaction data and demonstrate an autonomous optimization loop based on a Bandit algorithm.

## 4. Terminology

| Term | Definition |
|------|------|
| **State key** | An identifier representing the cat's current state, generated by combining audio classification, facial emotion, pose estimation, and CLIP scores |
| **User context** | Free-text information set by the cat's owner describing preferences, personality, or traits, used by Gemini when rewriting the prompt |
| **Template library** | A collection of video prompt templates from which Bandit selects, initially around 40 to 50 items |
| **Bandit table** | A data structure that manages cumulative reward and selection count for each template |
| **Reward** | A value based on the change in the cat's state after watching a video |
| **Experience Mode** | Demo mode where judges interact while "pretending to be the cat" |
| **Production Mode** | Intended real-use mode using actual cat data such as `.wav` and `.jpg` |
| **Pets in the Loop** | A loop concept in which cat reaction data and human annotation jointly improve the AI model |

## 6. Functional Requirements

### 6.1 Mode configuration

The app branches from the top screen into two modes: Experience Mode and Production Mode.

### 6.2 Experience Mode

Judges directly experience the product with the concept "You become the cat." Two input routes are prepared:

**Method A: Prepared samples**
- Select one of three recorded meow samples.
- Select one of three prepared face-photo samples.
- Choose or enter user context such as the cat's personality or preferences.

**Method B: Real-time input**
- Record a cat-like meow through the browser microphone API.
- Take a "cat-like face" photo through the browser camera API.
- If analysis fails, automatically fall back to Method A and notify the user.

### 6.3 Production Mode

- Upload `.wav` meow data and `.jpg` / `.png` images.
- Enter user context manually.
- Analyze the data, generate a state key, select a template, rewrite the prompt with Gemini, generate a video with Veo3, then record feedback.

### 6.4 Shared output screen

- Show the state key, selected template, and progress status.
- Play the generated video in full-screen or near-full-screen mode.
- Accept one of three feedback choices from a human observer:
  - `­¤ś║ Excited!`
  - `­¤śÉ So-so`
  - `­¤śŠ Not interested`

### 6.5 AI pipeline

#### Layer 1: Input

| Analysis target | Model | Output |
|---------|--------|------|
| Cat meow (`.wav`) | `IsolaHGVIS/Cat-Meow-Classification` | `brushing` / `waiting_for_food` / `isolation` |
| Cat facial emotion | `semihdervis/cat-emotion-classifier` | `happy` / `sad` / `angry` |
| Cat full-body pose | `usyd-community/vitpose-plus-plus-small` (ViTPose++) | 17 body keypoints ├Ś (`x`, `y`, `confidence`) = 51 dimensions |
| Zero-shot cat state classification | `openai/clip-vit-base-patch32` (CLIP) | Similarity scores to custom prompts |

> The logic for integrating the outputs of these four models into a state key will be finalized in the ML tuning phase.

#### Layer 2: Bandit

- Select the video template that should be shown now from a template library based on the generated state key.
- Candidate algorithm: UCB.
- Planned infrastructure: Vertex AI.

#### Layer 3: Generation

1. **Gemini (Vertex AI):** Rewrite the selected template into a cat-specific video-generation prompt using the state key and user context.
2. **Veo3 (GCP):** Generate a short video for the cat from that prompt.

#### Layer 4: Output

- Play the generated video full-screen in a Web browser.

#### Layer 5: Feedback

- A human inputs feedback after video playback.
- The three-choice reaction is converted into a reward score.
- Persist the Bandit table in Cloud Firestore.

#### Layer 6: Template evolution

- Automatically discard templates that continue to underperform the average.
- Gemini autonomously creates new templates to expand the library.

### 6.6 Data persistence and loops

#### Four loops

| Loop | Content | Priority |
|--------|------|------------|
| **Loop 1: In-session** | Cat watches a video, the reaction is scored, and the result is stored as reward | Required for MVP |
| **Loop 2: Cross-session learning** | Accumulated reward updates the Bandit table and improves future selection accuracy | Required for MVP |
| **Loop 3: Human annotation** | Humans create and label ground-truth data | Out of scope for this MVP |
| **Loop 4: Library evolution** | Low-rated templates are discarded and Gemini generates new ones | Best effort |

#### Data stored in Firestore

| Collection | Stored content |
|------------|---------|
| `sessions` | Session ID, input summary, state key, selected template ID, user context, timestamp |
| `bandit_table` | Template ID, cumulative reward, selection count, and last updated time |
| `templates` | Template ID, prompt template text, created time, active flag |
| `feedbacks` | Session ID, feedback value, converted reward score |

## 7. Non-functional Requirements

### Availability

- Keep the app running continuously during the hackathon demo window.
- Ensure Experience Mode can continue via fallback to sample selection.
- Show an error and retry option if Veo3 generation fails.

### Performance

- Target cat-state analysis within roughly 10 seconds.
- Veo3 generation may take time, so the UI shows a waiting screen.
- Gemini prompt generation should complete within a few seconds.

### Security

- No login is required for the MVP.
- GCP API keys and secrets are managed on the server side and never exposed to the frontend.
- Uploaded files are deleted after server-side processing when no longer needed.

### Maintainability and extensibility

- The template library is managed in Firestore and can be updated without redeployment.
- Cat-analysis models are deployed independently on Vertex AI Endpoint to simplify future replacement.

### Supported devices

- Primary target devices are PC and tablet in landscape orientation.
- The required browser is the latest Chrome.
- Smartphone support is desirable but not mandatory for the MVP.

```text

        |
        | HTTPS
        v
[Backend API]
        |
        +--> [Vertex AI Endpoint] <-> 4 HuggingFace models
        +--> [Vertex AI] Bandit model
        +--> [Gemini on Vertex AI] prompt rewriting and template evolution
        +--> [Veo3 on GCP] video generation
```

All four HuggingFace models are planned to be deployed to Vertex AI using a custom container or HuggingFace DLC.

## 11. Assumptions and Constraints

### Assumptions

1. GCP services such as Vertex AI, Veo3, Gemini, and Firestore are available.
2. The four HuggingFace models have licenses suitable for commercial and hackathon use.
3. The hackathon demo uses the latest Chrome.
4. Sample cat meows and photos for the demo are prepared in advance.

### Constraints

1. Development time, budget, and resources are limited.
2. Robotics hardware is difficult to procure, so the prototype is implemented as a Web app.
3. Veo3 generation takes time, so the UX must tolerate waiting.
4. Detailed algorithms and final I/O specifications will be finalized in a later ML tuning phase.
