## 1. プロジェクト概要

**プロダクト名：** nekkoflix  
**コンセプト名：** Pets in the Loop  
**形態：** Web アプリケーション（ハッカソンMVP）

### 一言説明

猫の状態（鳴き声・表情・ポーズ）をマルチモーダルAIで解析し、その猫に最適化された動画をリアルタイム生成して提供するWebアプリ。猫の反応データがモデルにフィードバックされ、「猫がAIを育てる」ループを実現する。

### ミッション

> 「人間と猫が一緒にAIに教えている」——  
> フィジカルAI普及前夜の現時点で、猫がAIと接するインタフェースを Web という形で先取り体験する。


### ストーリー

**起：**
「次はフィジカルAIだ」――その言葉が、あちこちで聞かれるようになった。ソフトウェアでできることはやり尽くした。次は、AIが身体を持つ時代だと。10〜20年後、ロボットが家庭に入り込むことはほぼ確実だろう。そのとき、家の中にいるのは誰だろうか。人間だけではない。猫といったペットもいる。

**承：**
私たちは初期のアイデア出し段階で、「人間×AI」のプロダクトを考えていた。Human in the Loop、AIとの協働、人間の体験を豊かにするツール……気づけば、すべての設計思想の中心に「人間」を置いていた。ふと思った。「動物のことを、完全に忘れていた。」AIというテクノロジーは、人間にとっては「使い慣れた道具」になりつつある。しかし動物にとってはどうだろうか。もっぱら将来の予測では「人間×AI」という文脈で議論されることが非常に多い。

**転：**
Brand Newの再解釈 「Brand New "Hello World."」――おそらくこのテーマのメインメッセージは「新しい常識を作ろう」というものだろう。我々はそこに以下の独自解釈を加えた。"Brand New"には「新品の」というニュアンスが包含される。AIはもう人間にとって「新品」ではないかもしれない。しかし動物にとっては、フィジカルAIの登場で人間以外も触れられるインタフェースの獲得を通じて、新しいAIへの Hello World になれるかもしれない。私たちの「Brand New Hello World.」は、動物がAIに初めて触れる瞬間を作ることとも言えるかもしれない。

**結：**
フィジカルAIが普及した未来、ペットはロボット（ハード）を通じてAIと日常的にインタラクションする。そのペットの反応が、モデルを育てる。ペットがループの中に入る。さらに、正解データの作成には人間によるアノテーションが不可欠であり、猫の反応フィードバックと人間の知識が組み合わさって初めてモデルは進化する。我々は **Pets in the Loop** という新しい概念をこのハッカソンを通じて提唱したい。human も animal も機能によっては協働して AI の発展に寄与することが、来るフィジカルAIの登場で可能になる。その未来を、現在のWebという形で先取り体験・現在の技術で頑張ってアプローチする。

## 2. 背景・目的

### 背景

- フィジカルAI（ロボティクス等）の普及が10〜20年後に予測されており、家庭内ペットとAIの共存が現実的になりつつある。
- 「Human in the Loop」が主流のAI開発において、**動物の視点が完全に欠落している**という問題提起。
- ハッカソンテーマ「Brand New "Hello World."」に対し、**動物にとってAIはまだ"Brand New"な存在**という独自解釈。

### 目的

1. 「猫専用AIエンターテインメント」という新しい概念をプロトタイプとして実証する。
2. ハッカソン審査員向けに、**"あなたが猫になる"体験**を通じてコンセプトを直感的に伝える。
3. 猫の反応データを蓄積し、Banditアルゴリズムによる自律最適化ループの動作をデモする。

## 4. 用語定義

| 用語 | 定義 |
|------|------|
| **状態キー** | 音声分類・顔感情・ポーズ推定・CLIPスコアを組み合わせて生成する、猫の現在状態を表す識別子 |
| **ユーザーコンテキスト** | 猫のオーナー（人間）があらかじめ設定する猫の好み・性格・特性等の自由記述情報。Geminiがクエリを再構築する際の追加材料として使用する |

| **体験モード** | 審査員が猫に"なりきって"操作するデモ用モード |
| **本番モード** | 実際の猫のデータ（.wav / .jpg）を入力として使う実運用想定モード |
| **Pets in the Loop** | 猫の反応データと人間のアノテーションが車の両輪となってAIモデルを改善するループ概念 |

## 6. 機能要件

### 6.1 モード構成

アプリはトップ画面から2つのモードに分岐する。

```
┌─────────────────────────────────────────┐
│              🐱 nekkoflix               │
│                                         │
│   [ 体験モード（あなたが猫になる）]      │
│   [ 本番モード（実際の猫データ入力）]    │
└─────────────────────────────────────────┘
```

---

### 6.2 体験モード（Experience Mode）

「あなたが猫になる」をコンセプトに、審査員が直接プロダクトを体験するモード。カメラの前で動くだけで自動的に撮影が行われる **3ステップの線形フロー** で構成される。サンプルカード選択・フォールバックは廃止済み。

#### Step 1：撮影（動き検知による自動キャプチャ）

- カメラが自動起動し、動き検知（`diff-cam-engine`）が開始される
- 猫のポーズをとったり、カメラの前で動くことで動き検知がトリガーされる
- 動き検知から1秒の安定待機を経て静止画を自動キャプチャする
- キャプチャ完了後、自動で Step 2 へ遷移する

#### Step 2：鳴きマネ録音（ボタン押下で3秒録音）

- 「録音を開始する（3秒）」ボタンが表示される
- 準備ができたらボタンを押すと、**3秒間**の音声録音が始まる（※ 3秒と画面に注記を表示）
- 「にゃあ」「みゃー」など猫の鳴き声を真似して録音する
- 録音完了後、自動で Step 3 へ遷移する（「スキップ」リンクで音声なしでも進める）

#### Step 3：コンテキスト入力

- 演じた猫の性格・好みを自由記述テキストエリアに入力する（任意・最大500文字）
- 例：「魚が大好きで、少し臆病な性格。窓の外の鳥によく反応する。」
- 空欄のまま生成することも可能（デフォルト設定で Gemini に渡す）
- 「動画を生成する」ボタンで生成リクエストを送信する

> **入力データ：** 静止画（base64）・音声（base64、任意）・コンテキストテキスト（任意）を `POST /generate` に直接送信する。GCS への事前アップロードは不要。

---

### 6.3 本番モード（Production Mode）

実際の猫データを使い、Banditループによる自律最適化を実現するモード。**2ステップの線形フロー** で構成される。

#### Step 1：コンテキスト入力

- 猫の性格・好みを自由記述テキストエリアに入力する（任意・最大500文字）
- 例：「魚が好き、臆病な性格、外を眺めるのが趣味」
- 「次へ」ボタンで Step 2 へ進む

#### Step 2：撮影・録音（動き検知による自動処理）

- カメラが自動起動し、動き検知（`diff-cam-engine`）が開始される
- 猫をカメラの前に置くと、動きを検知して静止画を自動キャプチャする
- 同時に3秒間の音声録音が自動で開始される
- 処理完了後、自動で `POST /generate` リクエストを送信し、結果画面へ遷移する

#### 結果画面（本番モード固有）：reaction video 録画

- 動画再生開始と同時に **reaction camera** が起動し、猫の反応映像を最大8秒間録画する
- 録画した reaction video を GCS へ direct upload（signed URL方式）する
- `POST /sessions/{session_id}/reaction` でバックエンドへ通知する
- バックエンドが reaction video を解析して報酬値を算出し、Banditテーブルを非同期更新する

> **入力データ：** 静止画（base64）・音声（base64）・コンテキストテキストを `POST /generate` に送信。reaction video のみ GCS 経由（signed URL）でアップロードする。

---

### 6.4 Output画面

体験モード・本番モードで表示が一部異なる。

#### 共通：ローディング表示

- ステップメッセージをアニメーション付きで表示する（「Veo3が動画を生成中です...」等）
- 想定生成時間の目安を表示する（「通常30秒〜3分かかります」）

#### 共通：動画再生

- 生成された動画を自動再生する
- ミュート・音量調整コントロールを提供する

#### 体験モード固有

- 動画をループ再生する
- 生成情報（テンプレート名・state_key）を表示する
- フィードバックボタンは**表示しない**

#### 本番モード固有

- 動画を1回再生する（ループなし）
- 動画再生開始と同時に **reaction camera** が起動し、猫の反応を最大8秒間録画する
- reaction video の upload 状態（録画中 / アップロード中 / 完了 / エラー）をリアルタイム表示する
- フィードバックボタンは**表示しない**（reaction video がその役割を代替する）

### 6.5 AIパイプライン

#### Layer 1：Input — 多峰性解析

| 解析対象 | モデル | 出力 |
|---------|--------|------|
| 猫の鳴き声（.wav） | `IsolaHGVIS/Cat-Meow-Classification` | `brushing` / `waiting_for_food` / `isolation` |
| 猫の顔感情 | `semihdervis/cat-emotion-classifier` | `happy` / `sad` / `angry` |
| 猫の全身ポーズ | `usyd-community/vitpose-plus-plus-small`（ViTPose++） | ボディキーポイント17点 × (x, y, confidence) = 51次元 |
| 猫の状態ゼロショット分類 | `openai/clip-vit-base-patch32`（CLIP） | カスタムプロンプト（例：`"interested cat"` 等）との類似度スコア |

> **📋 TBD：** 上記4つの出力を統合して状態キーを生成するロジックの詳細（重み付け・正規化・キー設計）は、MLチューニングフェーズで決定する。

#### Layer 2：Bandit — 最適化選択

- 生成された状態キーに対し、テンプレートライブラリ（初期40〜50本）から「今見せるべき動画テンプレート」を選択する
- アルゴリズム候補：UCB（Upper Confidence Bound）等
- インフラ：Vertex AI 上にデプロイ

> **📋 TBD：** 使用アルゴリズムの詳細・ハイパーパラメータは今後決定する。

#### Layer 3：生成 — AIオーケストレーション

1. **Gemini（Vertex AI）：** 選ばれたテンプレート＋状態キー＋**ユーザーコンテキスト（猫の好み・性格）**から、猫に最適化された動画生成プロンプトを再構築して生成
2. **Veo3（GCP）：** 生成プロンプトに基づき、猫向けの短編動画を生成

> **ユーザーコンテキストの役割：** 状態キーはBanditによるテンプレート選択に使用する一方、ユーザーコンテキストはGeminiがそのテンプレートを個別の猫に合わせてカスタマイズするための情報として機能する。例えば、同じ `T02 (playful yarn ball bouncing)` テンプレートが選ばれた場合でも、「魚が好きな猫」と「鳥が好きな猫」では異なるプロンプトが生成される。

#### Layer 4：Output

- 生成動画をWebブラウザ経由でフルスクリーン再生（猫の前に置いたスクリーンへ）

#### Layer 5：Feedback — 報酬計算（本番モードのみ）

- 動画再生開始と同時に猫の反応映像（reaction video）を最大8秒間録画する（`useReactionRecorder`）
- 録画した reaction video を GCS へ direct upload する（Backend 発行の signed URL 方式）
- `POST /sessions/{session_id}/reaction` で Backend へ `reaction_video_gcs_uri` を通知する
- Backend が reaction video を解析し、報酬値を算出して Bandit テーブルを非同期更新する

> **体験モードでは Feedback / 報酬計算は行わない。** モデル更新は本番モード専用。

#### Layer 6：テンプレート進化 — 自律改善

- 報酬が全体平均を継続的に下回るテンプレートを自動廃棄
- Gemini が新テンプレートを自律生成してライブラリを拡張

> **📋 TBD：** 廃棄閾値・新テンプレートの生成トリガー条件は今後決定する。

### 6.6 データ永続化・ループ

#### 4つのループ

| ループ | 内容 | 実装優先度 |
|--------|------|------------|
| **ループ①：セッション内ループ** | 猫が動画を視聴 → reaction video で反応を録画 → GCS upload → Bandit 更新 | MVP必須（本番モードのみ） |
| **ループ②：セッション間学習** | 蓄積報酬でBanditテーブルを更新 → 次回の選択精度が向上 | MVP必須 |
| **ループ③：人間によるアノテーション** | 正解データ作成・ラベリングは人間が担う | **スコープ外（今回は対象外）** |
| **ループ④：ライブラリの進化** | 低評価テンプレート廃棄 → Geminiが新テンプレートを自律生成 | ベストエフォート |

#### Firestoreに保存するデータ

| コレクション | 保存内容 |
|------------|---------|
| `sessions` | セッションID・入力データ概要・状態キー・選択テンプレートID・ユーザーコンテキスト・タイムスタンプ |
| `bandit_table` | テンプレートID・累積報酬・選択回数・最終更新日時 |
| `templates` | テンプレートID・プロンプトひな形・作成日時・有効フラグ |
| `feedbacks` | セッションID・フィードバック値（😺/😐/😾）・変換後報酬スコア |

## 7. 非機能要件

### 可用性

| 項目 | 要件 |
|------|------|
| ターゲット稼働率 | ハッカソン当日（数時間）の継続稼働を保証する |
| 障害時対応 | 体験モードは入力方法Aへのフォールバックを実装し、デモ継続を担保する |
| Veo3生成失敗時 | エラーメッセージ表示 + リトライボタンを提供する |

### 性能

| 項目 | 要件 |
|------|------|
| 猫状態解析（4モデル合計） | ユーザー体感でストレスのない応答を目標とする（目安：10秒以内）|
| Veo3動画生成 | リアルタイム生成のため、生成中は待機画面を表示して待つ設計とする（SLAは設定しない）|
| Geminiプロンプト生成 | 数秒以内を目標とする |

### セキュリティ

| 項目 | 要件 |
|------|------|
| 認証 | 不要（全画面オープンアクセス） |
| APIキー管理 | GCP APIキー等はサーバーサイド（バックエンド）で管理し、フロントエンドに露出させない |
| 入力ファイル | アップロードされたファイルはサーバーサイドで処理後、不要になり次第削除する |

### 保守性・拡張性

| 項目 | 要件 |
|------|------|
| テンプレートライブラリ | Firestore上で管理し、デプロイなしで追加・削除できる設計とする |
| モデル入れ替え | 猫分析モデルをVertex AI Endpointとして独立させ、将来の差し替えを容易にする |

### 対応デバイス

| 項目 | 要件 |
|------|------|
| 主対象デバイス | PC・タブレット（横向き）を主対象とする |
| ブラウザ | 最新版 Chrome を必須対応とする（デモ環境として使用） |
| レスポンシブ | スマートフォンは考慮するがMVP必須要件ではない |

## 8. システムアーキテクチャ概要

```
[ブラウザ（Next.js / React）]
        │
        │ HTTPS
        ▼
[バックエンドAPI（Next.js API Routes / サーバーレス）]
        │
        ├──► [Vertex AI Endpoint] ◄── HuggingFace モデル × 4
        │        Cat-Meow-Classification
        │        cat-emotion-classifier
        │        ViTPose++ (vitpose-plus-plus-small)
        │        CLIP (clip-vit-base-patch32)
        │
        ├──► [Vertex AI] ── Banditモデル
        │
        ├──► [Vertex AI / Gemini] ── プロンプト再構築（状態キー＋ユーザーコンテキスト）・テンプレート自律進化
        │
        ├──► [GCP / Veo3] ── 動画生成
        │
        └──► [Cloud Firestore] ── Banditテーブル・セッションデータ・ユーザーコンテキスト永続化
```

### HuggingFace モデルの Vertex AI デプロイ方針

調査の結果、**HuggingFaceの4モデルすべてをVertex AI Custom Container（Custom Prediction Routine / HuggingFace DLC）でデプロイすることは技術的に実現可能**であることを確認済み。

| 項目 | 方針 |
|------|------|
| デプロイ方式 | Vertex AI Custom Prediction Routine（CPR）または HuggingFace PyTorch Inference DLC |
| ベースイメージ | `huggingface-pytorch-inference-cu121`（HuggingFace 公式 DLC）|
| 初期構成 | CPUインスタンスで起動し、レイテンシを確認の上GPUへ移行を検討 |
| GPU対応 | Vertex AIはGPUインスタンス（NVIDIA T4等）に対応しており、必要に応じて有効化可能 |
| モデルの管理 | Vertex AI Model Registryに登録し、Endpoint経由でオンライン予測を実行 |

## 10. スコープ外（Out of Scope）

以下は今回のMVPスコープに**含まない**。

| 項目 | 理由 |
|------|------|
| 人間によるアノテーション管理UI（ループ③） | ループ構造の概念として定義するが、UIは不要。ペットのbefore/after差分をモデル改善に活用する設計とする |
| 認証・ログイン機能 | ハッカソンデモのためオープンアクセスで十分 |
| 猫以外の動物への対応 | MVPは猫に特化 |
| スマートフォン最適化 | デモ環境はPC/タブレット |
| ネイティブアプリ（iOS / Android） | Web Appで代替 |
| ハードウェア（ロボット等） | コスト・調達困難のため今回は対象外 |
